# Cohesion

A buildable narrative is not a tree. It is a graph.

The hierarchy (Domain → Narrative → Scene → Moment) gives the structure. The cohesion underneath gives the meaning.

## The hierarchy is the visible part

When you first see a narrative, you see a tree:

```text
Concert Booking
├── A fan gets tickets
│   ├── Tickets reserved
│   ├── Booking blocked because show is sold out
│   ├── Booking cancelled
│   └── Fan added to waitlist
└── A promoter cancels a show
    └── Show cancelled
```

Each scene contains moments. Each moment captures a step. The hierarchy is where structure lives.

But the hierarchy alone is not enough.

## The cohesion is the graph

Underneath the tree, the moments and scenes connect through cause and effect.

A command in one moment produces an event. The event triggers a react moment in another scene. A query in a third moment reads state that the command updated.

For example:

```text
Booking cancelled (Scene)
  CancelBooking [Command]
    BookingCancelled [Event]
      triggers → Fan promoted from waitlist (React in Waitlist scene)
        consumes → produces NewReservation
```

That cross-reference is the cohesion. Without it, the cancellation scene and the waitlist scene look like neighbors that happen to share vocabulary. With it, they share a real chain of cause and effect that the model makes explicit.

## Three kinds of cohesion

### Event cohesion

Commands in one scene produce events. React moments in other scenes consume them.

This is the spine of system behavior. It is what makes a cancellation actually trigger a waitlist promotion, instead of two separate features that happen to use the same words.

### Data cohesion

Visible data traces back through events to the command that produced it.

```text
Booking confirmation screen displays "Booking Reference"
  → Reference field on Booking entity
    → Generated by ReserveTickets command
      → Initiated by Reserve Tickets moment in Tickets reserved scene
```

This is data completeness, expressed as a chain. (See [Data Completeness](/explanation/data-completeness) for the longer treatment.)

### Reference cohesion

Moments can reference other moments directly. A query moment that displays "Available shows" depends on the publish-show command in a different narrative. The list and the source are linked, not duplicated.

This catches a class of bugs where the same concept is modeled twice in slightly different ways across scenes.

## Why cohesion matters for AI

A coding agent receives a slice of the narrative. The slice has structure. Without cohesion, the slice is missing the references it needs.

If the cancellation scene has a CancelBooking command and the waitlist scene has a "promote next fan" react, but no link between them, the agent has to invent the connection. It will guess based on naming. Sometimes correctly, sometimes not.

Cohesion makes the connection explicit. The agent does not guess.

## Why cohesion matters for humans

A reviewer reading a single scene cannot tell whether the scene is consistent with the rest of the narrative.

A reviewer reading the cohesion graph can see:

- which events the scene produces
- which scenes consume those events
- where the data the scene displays comes from
- which other moments depend on this scene's behavior

This is what catches broken flows during review, before the code exists.

## Story mapping vs NDD

User story mapping gives you the tree of journeys, scenes, and steps.

NDD adds the graph: the cross-references between moments, the events that travel across scenes, the data that flows through the structure.

The tree organizes the work. The graph keeps it coherent.

## What to check

When you review a narrative, look for:

1. Every event has at least one producer and at least one consumer.
2. Every visible field has a traceable source.
3. Every cross-scene dependency is explicit, not implied.
4. Every "this triggers that" is captured as a real reference, not just naming.

If a connection is missing, the model is incomplete, even when the hierarchy looks fine.
